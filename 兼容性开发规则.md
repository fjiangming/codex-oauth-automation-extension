# 项目级 AI 开发规则

> **本规则是对所有 AI（包括但不限于 Gemini、Claude、Copilot、Cursor）生成代码时的强制性约束。**

---

## 🎯 核心原则：Linux CentOS 9 Stream x64 兼容性

本项目的所有代码、脚本、配置均必须能在 **Linux CentOS 9 Stream x86_64** 上兼容运行。这不是建议，是硬性约束。

### 目标运行环境

| 项目 | 规格 |
|------|------|
| 操作系统 | CentOS 9 Stream x86_64 |
| 内核 | 5.14+ (el9) |
| 包管理器 | dnf (yum 兼容) |
| 默认 Shell | bash 5.1+ |
| Python | python3 (3.9+, 系统自带) |
| Node.js | 如需使用，优先通过 `dnf module install nodejs` 安装 |
| Init 系统 | systemd |
| 进程守护 | Supervisor 4.x |
| SELinux | 默认 enforcing，代码不得假设 SELinux 已禁用 |

---

## Shell 脚本规范

### 必须遵守

1. **Shebang 行**：一律使用 `#!/bin/bash`，不使用 `#!/bin/zsh`、`#!/bin/sh`（避免 dash 不兼容）或 `#!/usr/bin/env bash`（CentOS 9 的 bash 固定在 `/bin/bash`）。
2. **Bash 兼容语法**：仅使用 Bash 4.x/5.x 支持的语法，不依赖 zsh/fish 专有特性。
3. **命令可用性**：只使用 CentOS 9 默认安装或通过 `dnf` 可安装的命令。常用命令白名单：
   - 文件操作：`cp`, `mv`, `rm`, `mkdir`, `chmod`, `chown`, `find`, `ls`, `cat`, `head`, `tail`, `wc`, `stat`, `rsync`
   - 文本处理：`grep`, `sed`, `awk`, `cut`, `sort`, `uniq`, `tr`, `xargs`, `tee`
   - 网络工具：`curl`, `wget`, `ss`（不使用 `netstat`，已在 CentOS 9 中弃用）
   - 进程管理：`ps`, `kill`, `pgrep`, `pkill`, `nohup`, `supervisorctl`
   - 系统管理：`systemctl`, `journalctl`, `dnf`
   - 其他：`date`, `sleep`, `python3`, `jq`（需显式安装）
4. **路径约定**：
   - 不硬编码 macOS / Windows 路径（如 `/Users/`, `C:\`）。
   - 使用 `$HOME` 而非 `~` 在变量赋值中（`VAR=~/foo` 在某些上下文不展开）。
   - 可执行文件优先使用绝对路径或 `command -v` 检查。
5. **换行符**：所有脚本和配置文件必须使用 **LF**（`\n`），不能有 **CRLF**（`\r\n`）。Windows 环境开发时注意 `.gitattributes` 或 `dos2unix` 转��。
6. **编码**：所有文件一律 **UTF-8 无 BOM**。

### 禁止使用

- `brew`、`apt`、`apt-get`、`pacman`、`apk` 等非 RHEL 系包管理器命令。
- `pbcopy`/`pbpaste`、`open`（macOS 专有）。
- `netstat`（CentOS 9 已弃用，使用 `ss` 替代）。
- `ifconfig`（已弃用，使用 `ip` 命令替代）。
- `readlink -f` 可用，但如需更安全可用 `realpath`。
- GNU 以外的 `sed`/`awk`/`grep` 行为假设（如 macOS 的 BSD sed 语法）。

## Node.js / JavaScript 规范

### 必须遵守

1. **文件路径**：使用 `path.join()` / `path.resolve()` 拼接路径，不硬编码 `\` 或 `/`。
2. **换行符处理**：解析文本时使用 `\r?\n` 正则，不假设行尾一定是 `\n`。
3. **编码**：文件读写明确指定 `utf-8`。
4. **子进程调用**：
   - 使用 `child_process.spawn` / `execFile` 时，shell 路径使用 `/bin/bash`。
   - 不依赖 Windows `cmd.exe` 或 PowerShell 语法。
5. **原生模块**：如果引入需要编译的 npm 原生模块（如 `node-gyp`），必须确认 CentOS 9 上 `gcc`/`g++`/`make` 的可用性，并在文档中注明。
6. **文件权限**：如果脚本需要执行权限，代码中应通过 `fs.chmodSync(path, 0o755)` 显式设置，不依赖 Windows 的权限模型。

### 禁止使用

- Windows 注册表 API。
- Windows 特有的路径分隔符假设（如 `C:\\Users\\...`）。
- 未经验证的仅在 macOS/Windows 上可用的 npm 包。

---

## Python 脚本规范

### 必须遵守

1. 使用 `#!/usr/bin/env python3` 或 `python3` 调用，不使用 `python`（CentOS 9 默认无 `python` 命令，只有 `python3`）。
2. 优先使用标准库，不依赖需要 `pip install` 的第三方包，除非在文档中明确注明安装步骤。
3. 文件操作使用 `pathlib` 或 `os.path`，不硬编码路径分隔符。

---

## 配置与部署规范

### 必须遵守

1. **Supervisor 配置**：
   - 配置文件放在 `/etc/supervisord.d/` 目录，扩展名 `.ini`。
   - 不使用 systemd unit 文件替代 Supervisor（项目统一使用 Supervisor 管理进程）。
2. **防火墙**：如果新增网络端口，必须在文档中注明需要通过 `firewall-cmd` 开放端口。不假设防火墙已关闭。
3. **SELinux**：
   - 如果文件路径不在标准位置（如 `/opt`、`/var`、`/home`），可能需要 `semanage` 或 `restorecon` 设置上下文。
   - 不在代码中执行 `setenforce 0` 或建议用户禁用 SELinux。
4. **日志路径**：使用项目约定的 `/var/log/codex-oauth-automation/` 目录，不写到 `/tmp` 或用户家目录。
5. **临时文件**：使用 `mktemp` 创建，不硬编码 `/tmp/xxx`。
6. **Cron 定时任务**：如需定时任务，使用 `systemd timer` 或 `crontab`，不依赖第三方定时方案。

---

## 软件包安装规范

### 必须遵守

1. 安装命令一律使用 `dnf install -y <包名>`。
2. 如果需要 EPEL 仓库中的包，必须先 `dnf install -y epel-release`。
3. 不使用 `yum`（虽然 CentOS 9 中 yum 是 dnf 的别名，但为统一性始终使用 `dnf`）。
4. 如果需要的软件不在官方仓库，必须提供明确的 RPM 安装或编译步骤。

## AI 自检清单

每次生成涉及系统交互的代码后，必须自问：

1. ✅ 这段脚本在 CentOS 9 Stream 的 Bash 5.1 下能否正常运行？
2. ✅ 使用的命令是否在 CentOS 9 默认安装或 dnf 可安装范围内？
3. ✅ 是否有 macOS / Windows / Ubuntu 专有语法或命令混入？
4. ✅ 文件路径是否跨平台安全（无硬编码 `\`、无 Windows 盘符）？
5. ✅ 换行符是否为 LF？编码是否为 UTF-8 无 BOM？
6. ✅ 是否考虑了 SELinux enforcing 模式的影响？
7. ✅ 是否使用了 `dnf` 而非 `apt` / `brew` / `pacman`？
8. ✅ Python 调用是否使用了 `python3` 而非 `python`？
9. ✅ 网络命令是否使用了 `ss` / `ip` 而非 `netstat` / `ifconfig`？
10. ✅ 新增端口是否在文档中注明了 `firewall-cmd` 开放步骤？

---

## 违反处理

如果生成的代码不满足上述任何一条约束：

- **视为未完成**，需修正后重新提交。
- 不接受"在其他系统上能运行"作为理由。
- 唯一的兼容性判断标准是：**能否在 CentOS 9 Stream x86_64 上正确执行**。
